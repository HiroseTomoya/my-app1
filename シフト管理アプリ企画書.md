# シフト管理アプリ

シフトの提出から確定・確認までをひとつのアプリで完結させ、従業員と管理者の双方のストレスを軽減するシフト管理アプリケーションです。

## ターゲットユーザー
- **従業員（アルバイト・社員）**: シフトを提出し、確認するユーザー
- **シフト管理責任者**: 従業員の希望を取りまとめ、シフトを作成・確定するユーザー

---

## 抱えていた課題

### 従業員側の課題
* **プロセスの分断**: LINEで希望を提出し、Googleスプレッドシートで確認するというフローのため、一貫性がなく分かりづらい。
* **変更箇所の把握が困難**: 提出した「希望シフト」と実際の「確定シフト」を見比べるのが難しく、どこが削られたのか直感的に分からない。
* **スマホでの視認性の悪さ**: スプレッドシートはスマホだと画面の一部しか見えず、一覧性がなく使い勝手が悪い。
* **デザイン性**: 既存の運用はUI/UXの観点から使いづらく、見た目が良くない（※最も重要な課題）。

### 管理者側の課題
* **集計時の手間の多さ**: 従業員から送られてくる個々のLINEメッセージを開きながらシフトを作成しなければならず、非常に手間がかかる。

---

## ソリューション

| 課題 | 当アプリでの解決策 |
| :--- | :--- |
| プロセスの分断 | **アプリへの一元化**<br>シフト提出から確認までを1つのアプリにまとめ、一貫性を持たせます。 |
| 変更箇所の把握 | **タブ切り替え比較**<br>「提出したシフト」と「実際のシフト」をタブで切り替えられるカレンダー形式を採用し、差分を分かりやすくします。 |
| スマホでの視認性 | **最適化されたカレンダーUI**<br>自分のシフトをカレンダーで全体把握できるようにし、別画面で「全員のシフト」もカレンダー形式で快適に閲覧できるようにします。 |
| デザイン性 | **シンプルでモダンなUI**<br>日常的に使うものだからこそ、シンプルで誰もが見やすく使いやすいデザインを追求します。 |
| 管理者の手間 | **直感的なシフト編集機能**<br>スキルバランスの考慮が必要なため自動作成は行わず、日ごとに「誰が希望しているか」を一覧表示。名前をタップして時間を編集し、確定すると即座に従業員側に反映されるUIを提供します。 |

## コア機能 / MVP
リリース時に絶対に外せない3つのコア機能：

1. **タブ切り替え式 シフト確認カレンダー**
   * 自分が提出したシフトと、実際の確定シフトをタブで瞬時に切り替えて比較・確認できる機能。
2. **かんたんシフト提出機能** 
   * スマホから直感的に希望シフト（時間帯）を入力・送信できる機能。
3. **管理者向け シフト編集・確定機能** 
   * 日付ごとに希望者を一覧表示し、タップで勤務時間を調整、確定ボタンで即座に従業員へ反映させる機能。
4. **全体のシフト確認カレンダー**
   * 全員のシフトが全体的に見ることができる一覧性のある機能

## 機能仕様
「もし自分がこのアプリのユーザーだったら、ここでどう動いてほしいか？」を徹底的にシミュレーションする:

### [1] 「画面要素」と「ユーザーの行動」をすべて書き出す
**▼ 共通・従業員（マイページ）**
*   **ログイン画面を開いたとき**：ユーザーIDとパスワードを入力すると自分のマイページに飛べるようにする。
*   **マイページを開いたとき**：「マイシフト」「全体シフト」「シフト提出」の画面を上のメニューバーでそれぞれ選択できるようにする。
*   **「マイシフト」を開いたとき**：「希望シフト」と「確定シフト」をタブで瞬時に切り替えて比較・確認できる。それぞれ祝日が表示されるカレンダー形式で表示される。その時、全体ではシフトの時間のみ表示されるようにする
*   **「確定シフト」タブで特定の日付をタップしたとき**：その日のポジション（ホール・キッチンなど）と、休憩時間・実働時間などの詳細情報が分かるようにする。
*   **「全体シフト」を開いたとき**：全員のシフトがカレンダー形式で一覧表示される。カレンダー全体ではホール、ドリンカー、社員、バイトリーダーなどで色分けされた名前のみをを表示し、日付をタップすると、その日に出勤する全メンバーの名前と勤務時間帯、ポジションなどががリスト表示される。
*   **「シフト提出画面」を開いたとき**：次月のカレンダーが表示され、日付をタップして勤務可能時間（例：17:00〜23:00、または「フリー」）を直感的に入力できる。また、一括で入力できる機能も追加する。最後に入力内容を確認し、「提出する」ボタンを押すと完了する。
**▼ 管理者（シフト作成）※管理者権限のアカウントのみ表示**
*   **「シフト作成画面」を開いたとき**：日付ごとのカレンダーが表示され、各日付に「希望を出している人数」が表示される。
*   **特定の日付をタップしたとき**：その日にシフト希望を出している従業員が色分けされて一覧表示される。
*   **従業員の名前をタップしたとき**：その人の勤務時間、ポジション、休憩時間を直感的に編集できる画面が開く。その時、時間の編集画面では希望シフトの時間（フリー等）が初期設定として反映されるようにする
*   **「シフトを確定する」ボタンを押したとき**：編集内容が保存され、従業員側の「確定シフト」および「全体シフト」に即座に反映される。もう一度修正したい場合でも修正できるようにする

---

### [2] 「データのルール（制約）」を決める
*   **権限の区別**：「一般従業員」と「管理者」の2つのアカウント権限が存在する。一般従業員はシフト作成・編集画面にはアクセスできない。
*   **入力の単位**：シフトの時間入力は「15分単位（または30分単位）」のプルダウン等にし、細かすぎる入力ブレを防ぐ。「フリー」という選択肢も可能にする。
*   **シフト提出期限**：毎月〇日（管理者が設定可能）を過ぎると、従業員側からは該当月のシフト提出・変更ができなくなる（ロックがかかる）。
*   **重複登録の防止**：同じ日付に対して、1人の従業員が2つ以上の異なる希望シフトを重複して送信できないようにする。
*   **必須項目**：シフト提出時は必ず「開始時間」と「終了時間」の両方がセットで入力されていなければエラーを出す（「フリー」選択時は除く）。

---

### [3] 「見えない裏側の動き（データ保存）」を決める
*   **ユーザー情報（アカウント作成時）**
    *   ユーザーID、暗号化されたパスワード、氏名、権限（一般 or 管理者）、デフォルトのポジション等のデータが `Users` テーブルに保存される。
*   **シフト提出時（希望シフトの保存）**
    *   従業員が「提出」ボタンを押した瞬間に、「誰が（ユーザーID）」「いつ（日付）」「何時から何時まで」希望したかというデータが `Desired_Shifts` テーブルに保存される。
*   **シフト作成・確定時（確定シフトの保存）**
    *   管理者が時間を編集して「確定」を押すと、その内容（決定した時間、休憩時間、ポジション）が `Confirmed_Shifts` テーブルに保存・更新される。
    *   従業員がマイページを開いた際、この `Confirmed_Shifts` テーブルから最新のデータが取得（フェッチ）され、画面のカレンダーに描画される。
